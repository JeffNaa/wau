Wau Core 文档

Wau Core 是一个基于 NestJS 的插件运行时引擎。它动态加载、管理和路由第三方插件，同时通过 Prisma 将插件元数据持久化到 PostgreSQL 数据库。

技术栈

层级技术
框架NestJS 11
数据库PostgreSQL
ORMPrisma 7 + @prisma/adapter-pg
HTTPExpress（通过 NestJS platform-express）
配置@nestjs/config
工具库fs-extra, adm-zip

模块详解

1. PrismaModule src/prisma/

标记为 @Global()，因此 PrismaService 可以在任何地方注入，无需显式导入。

  • PrismaService 通过 pg.Pool 从 DATABASE_URL 创建 PrismaClient。
  • 实现 OnModuleDestroy，在关闭时优雅断开连接。
  • 暴露 client 属性直接访问模型（无仓库模式）。

2. PluginLoaderModule src/plugins/plugin-loader.module.ts

一个带有静态 forRoot(pluginsDir) 工厂方法的动态模块。它在应用启动时运行（此时依赖注入尚未完全就绪），因此扫描文件系统。

职责：

  • 枚举 storage/plugins/<name>/dist/index.js
  • 使用 Reflect.getMetadata('controllers', PluginModule) 检测控制器路径
  • 检测并拦截跨插件重复的控制器路由
  • 将有效插件模块导入 NestJS 模块图
由于该模块在静态工厂中运行，无法以数据库作为数据源。它保持基于文件系统的实现。

插件生命周期和运行时路由的核心引擎。

4. PluginController src/plugin.controller.ts

插件管理的 REST API 接口。基础路径：/api/plugins

方法路径请求体操作
POST/api/plugins/uploadmultipart file (ZIP)安装插件
GET/api/plugins—列出已安装插件
PUT/api/plugins/:namemultipart file (ZIP)按名称更新插件
DELETE/api/plugins/:name—按名称卸载插件

数据库模型

文件：prisma/schema.prisma

PluginRegistry

已安装插件元数据的权威来源。

model PluginRegistry {
  id        String   @id @default(cuid())
  name      String   @unique
  version   String
  manifest  Json
  mode      String   @default("jsonb")
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  @@map("plugin_registry")
}
字段类型说明
idCUID主键
nameString唯一；与插件目录名一致
versionStringmanifest 中的语义化版本
manifestJSONB完整的 manifest.json 内容
modeString预留，用于未来插件类型区分

PluginData

插件运行时键值数据存储。

model PluginData {
  id        String   @id @default(cuid())
  plugin    String
  key       String
  data      Json
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  @@unique([plugin, key])
  @@index([plugin])
  @@map("plugin_data")
}

插件格式规范

一个有效的 Wau 插件是一个 ZIP 压缩包，包含以下结构：

myplugin.zip
├── manifest.json          # 必需
├── package.json           # 必需（peerDependency: @nestjs/common）
├── tsconfig.json          # 必需
├── index.ts               # 入口文件：导出 NestJS 模块类
├── myplugin.module.ts     # NestJS @Module()
├── myplugin.controller.ts # NestJS @Controller() 路由
└── myplugin.service.ts    # 可注入服务

manifest.json

{
  "name": "testplugin",
  "version": "1.0.1",
  "description": "Test Plugin",
  "author": "Jeff"
}

必需字段：name、version。

构建输出

TypeScript 编译后，插件必须暴露 dist/index.js 作为入口文件。该文件必须导出 NestJS 模块类：

export { TestpluginModule } from './testplugin.module';

PluginRegistry 数据库迁移

变更内容

迁移前，PluginManagerService 仅依赖内存中的 Map 和文件系统扫描 manifest.json。虽然 PluginRegistry Prisma 模型已存在于 schema 中，但应用代码从未使用它。

新数据流

安装/更新/卸载
        |
        v
+-------------------+       +------------------+
| PluginManagerSvc  |<---->| plugin_registry  |
|  - 验证 ZIP       |  CRUD |  (PostgreSQL)    |
|  - 解压到文件系统 |       +------------------+
|  - 加载路由       |              |
|  - 持久化到数据库 |              v
+-------------------+       +------------------+
                            | storage/plugins  |
                            |  (代码文件)      |
                            +------------------+

一致性策略

场景行为
数据库有记录，代码存在启动时正常加载
数据库有记录，代码缺失警告并跳过（孤立的数据库记录）
代码存在，数据库无记录PluginManagerService 忽略；PluginLoaderModule 仍会加载 NestJS 模块。重新安装可补录数据库记录。

回滚行为

操作失败时的回滚
安装（路由冲突）删除已解压目录 + 删除数据库记录
更新（路由冲突）删除新解压的目录。旧代码已被删除；数据库已更新到新版本。这是架构限制。
卸载先删除数据库记录，再删除目录。若目录删除失败，文件系统残留孤儿文件。
