Wau Core Documentation

Wau Core is a NestJS-based plugin runtime engine. It scans and loads third-party plugin modules at boot time via PluginLoaderModule, while persisting plugin metadata in a PostgreSQL database via Prisma. The server auto-restarts after plugin install/update/uninstall to reload routes.

Module Breakdown

1. PrismaModule src/prisma/

Marked @Global() so PrismaService is injectable everywhere without explicit import.

  • PrismaService creates a PrismaClient backed by a pg.Pool from DATABASE_URL.
  • Implements OnModuleDestroy to gracefully disconnect on shutdown.
  • Exposes client for direct model access (no repository pattern).

2. PluginLoaderModule src/plugins/plugin-loader.module.ts

A dynamic module with a static forRoot(pluginsDir) factory. It runs at application bootstrap time (before DI is fully ready) and therefore scans the filesystem.

Responsibilities:

  • Enumerate storage/plugins/<name>/dist/index.js, falling back to storage/plugins/<name>/index.js if dist/ is absent
  • Use Reflect.getMetadata('controllers', PluginModule) to detect controller paths
  • Auto-prefix routes with manifest name (e.g. /test-plugin/status)
  • Detect and throw on duplicate controller routes across plugins
  • Import valid plugin modules into the NestJS module graph
Because this module runs in a static factory, it cannot use the database as its source of truth. It remains filesystem-based. After install/update/uninstall, the server auto-restarts to reload routes.

Database Schema

Files: prisma/schema/schema.prisma (generator & datasource) + prisma/schema/plugin.prisma (models)

PluginRegistry

Authoritative source of installed plugin metadata.

model PluginRegistry {
  id                String   @id @default(cuid())
  name              String   @unique
  version           String
  manifest          Json
  migrationsApplied Json?    @map("migrations_applied")
  createdAt         DateTime @default(now()) @map("created_at")
  updatedAt         DateTime @updatedAt @map("updated_at")

  @@map("plugin_registry")
}
FieldTypeNotes
idCUIDPrimary key
nameStringUnique; matches plugin directory name
versionStringSemver string from manifest
manifestJSONBFull manifest.json content
migrationsAppliedJSONBList of applied SQL migration filenames

PluginData

Key-value store for plugin runtime data.

model PluginData {
  id        String   @id @default(cuid())
  plugin    String
  key       String
  data      Json
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  @@unique([plugin, key])
  @@index([plugin])
  @@map("plugin_data")
}

Plugin Format Specification

A valid Wau plugin is a ZIP archive containing:

myplugin.zip
├── manifest.json          # Required
├── package.json           # Required (peerDependency: @nestjs/common)
├── tsconfig.json          # Required
├── index.ts               # Entry point: exports the NestJS Module class
├── myplugin.module.ts     # NestJS @Module()
├── myplugin.controller.ts # NestJS @Controller() with routes
├── myplugin.service.ts    # Injectable services
└── migrations/            # Optional: SQL migration files (.sql)

manifest.json

{
  "name": "test-plugin",
  "version": "1.0.1",
  "description": "Test Plugin",
  "author": "Jeff",
  "schema": {
    "products": {
      "sku": { "type": "string", "required": true, "unique": true },
      "name": { "type": "string", "required": true },
      "price": { "type": "decimal", "required": true, "precision": 10, "scale": 2 }
    }
  }
}

Required fields: name, version. Optional: schema (JSON schema for dynamic tables), description, author.

JSON Schema

Plugins can declare a schema object in manifest.json to have the core automatically create PostgreSQL tables. Table names are prefixed plugin_{name}_{model}. Each field supports type, required, default, unique, index, primary, length, precision, scale.

SQL Migrations

Plugins can include a migrations/ directory with .sql files. Files are executed in alphabetical order on install/update and tracked in plugin_registry.migrations_applied. On uninstall, tables prefixed plugin_{name}_* are dropped automatically.

Built Output

After TypeScript compilation, the plugin should expose dist/index.js as its entry point; if dist/ is absent, it falls back to root-level index.js. This file must export the NestJS module class:

export { TestPluginModule } from './test-plugin.module';

PluginRegistry Database Migration

What Changed

Prior to this migration, PluginManagerService relied solely on an in-memory Map and filesystem scanning of manifest.json files. The PluginRegistry Prisma model existed in the schema but was never used by application code.

New Data Flow

Install/Update/Uninstall
        |
        v
+-------------------+       +------------------+
| PluginManagerSvc  |<---->| plugin_registry  |
|  - Validate ZIP   |  CRUD |  (PostgreSQL)    |
|  - Extract to FS  |       +------------------+
|  - Trigger restart|            |
|  - Persist to DB  |              v
+-------------------+       +------------------+
                            | storage/plugins  |
                            |  (Code files)    |
                            +------------------+

Consistency Strategy

ScenarioBehavior
DB record exists, code existsNormal load at startup
DB record exists, code missingWarn and skip (orphaned DB record)
Code exists, DB record missingIgnored by PluginManagerService; PluginLoaderModule still loads NestJS module. Re-install to create DB record.

Rollback Behavior

OperationRollback on Failure
Install (route conflict)Delete extracted directory + delete DB record
Update (route conflict)Delete new extraction. Old code already deleted; DB updated to new version. This is an architectural limitation.
UninstallDB record deleted first, then directory. If directory deletion fails, filesystem orphan remains.
